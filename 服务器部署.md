# 内网服务器部署说明（Windows）

本文说明将本 OCR 微服务拷贝到**无外网**的 Windows Server 2016 / Windows 10 / Windows 11 服务器后，如何安装依赖、启动服务，以及配置**开机自动启动**。

> **说明（Windows）**：`gunicorn` 在 Windows 上不可用，生产与测试请使用项目根目录下的 **`startupV4m.bat` / `startupv5m.bat` / `startupv5s.bat`**（任选其一，对应不同模型组合），其内部使用 `uvicorn` 多进程承载并发，与 Linux 下 “多 worker” 的思路一致。

---

## 1. 需要拷贝到服务器的内容

建议**整体拷贝**本仓库（或发布目录），至少包含下列路径（相对项目根目录）：

| 路径 | 用途 |
|------|------|
| `app/` | 服务代码 |
| `gunicorn_conf.py` | 仅作 worker 数等参数参考（Linux 可用）；Windows 以环境变量 + 根目录启动脚本为准 |
| `requirements.txt` | 依赖声明（含 `paddlepaddle`，与锁定清单一致） |
| `offline_bundle/models/` | PaddleOCR 检测/识别/方向模型（程序默认相对路径读取） |
| `offline_bundle/paddle/` | `paddlepaddle` CPU 及传递依赖的离线 wheel |
| `offline_bundle/wheels/` | 业务依赖离线包（含 `requirements-lock.txt` 锁定清单） |
| `安装程序/` | Python 安装包、`VC++` 运行库等（若服务器尚未安装） |
| `startupV4m.bat` / `startupv5m.bat` / `startupv5s.bat` | 本机启动脚本（见第 5 节） |
| `uvicorn_log_config.json` | uvicorn 终端日志格式（含时间戳；与启动脚本 `--log-config` 配套） |
| `服务器部署.md` | 本文档 |
| `CSharp-IIS-调用示例.md` | （可选）C# 调用参考 |
| `scripts/verify_offline_bundle.py` | **外网拷贝到内网前自检**（模型文件是否齐全、`pip` 能否仅用本地 wheel 满足锁定清单） |
| `scripts/prepare_offline_assets.ps1` | **外网一键准备**离线 wheel、Paddle 目录与模型；收尾**自动执行** `verify_offline_bundle.py`，失败则不应拷贝 |

**不必拷贝**（可在服务器上重新生成）：

- `.venv/`：在服务器上创建虚拟环境即可  
- `.venv_offline_build/`：仅用于外网打包依赖，可不带

**不要依赖**打包机上的 `D:\Python313\Lib\site-packages`。服务器安装应**只使用** `offline_bundle` 中的 wheel，并以 `requirements-lock.txt` 为准。

## 2. 服务器环境准备

1. 安装 **Microsoft Visual C++ Redistributable（x64）**  
   可使用随包提供的 `安装程序\vc_redist.x64.exe`（或服务器已安装同等运行库）。

2. 安装 **与离线 wheel 匹配的 Python x64**  
   看 `offline_bundle\wheels` 下文件名：含 **`cp313`** 表示 **Python 3.13**，含 **`cp311`** 表示 **Python 3.11**。服务器上的 Python 主版本**必须一致**，否则离线 `pip install` 会失败。  
   `安装程序` 目录下的安装包仅作示例；若与你当前的 `cp3xx` 不一致，请在外网下载**同版本** Python 安装包放进 `安装程序` 再拷贝到内网，或重新用目标 Python 版本打 wheel。

3. 建议勾选 “Add Python to PATH”，或在后续命令中使用 Python 的完整路径。

4. **平台与当前仓库离线包一致**  
   本仓库已纳入的 wheel 为 **Windows x64**，且文件名含 **`cp313`** 时需使用 **Python 3.13 64 位**。若内网服务器为其他 Python 主版本或操作系统，须在外网用目标环境重新执行 `scripts\prepare_offline_assets.ps1` 生成对应 `offline_bundle`，**不可**混用不同 `cp3xx` 的 wheel。

---

## 3. 离线包完整性（核查结论与维护约定）

以下内容对应近期对 `offline_bundle` 的核查与修复，便于外网打包与内网验收对齐。

### 3.1 锁定清单与 wheel 目录

- 唯一依赖来源：`offline_bundle\wheels\requirements-lock.txt`（**UTF-8 编码**，供 `pip` 解析）；须与 `offline_bundle\wheels\*.whl` **一一对应**，缺一不可。
- 清单已包含 **`paddlepaddle`** 及 CPU 运行常见传递依赖（如 **`protobuf`、`opt-einsum`、`networkx`、`safetensors`** 等）。若曾只更新清单而未补 wheel，内网会报错；**已补全的 wheel 须随仓库或发布目录一并拷贝**。
- `offline_bundle\paddle\`：用于「先装 Paddle 栈再装清单」的两步安装；其中 **`pip download paddlepaddle`** 会带齐传递依赖。**勿**在同一目录保留与锁定版本冲突的重复包（例如多份不同版本的 `numpy` / `pillow`），以免误导排查。

### 3.2 模型目录

- 三套模型目录下均须存在 **`inference.pdmodel`** 与 **`inference.pdiparams`**（仅存在占位 `pdmodel` 无法运行）。外网可用 `scripts\download_models.py` 拉取；若目录已存在但不完整，脚本会删除后重下（见脚本逻辑）。

### 3.3 拷贝到内网前自检（外网执行）

```bat
cd /d E:\paddleOcr
python scripts\verify_offline_bundle.py
```

- 亦可直接运行 **`scripts\prepare_offline_assets.ps1`**，其**最后一步**会调用上述校验；失败则**不要**将 `offline_bundle` 拷入内网。
- 自检通过仅代表：模型文件齐全，且在当前机器上 **`pip install --no-index` + 双 `--find-links`** 能解析锁定清单；内网仍需 **Python 与 `cp3xx`、系统架构**与 wheel 一致（见上文第 2 节第 4 点）。

## 4. 离线安装 Python 依赖（在服务器上执行一次）

打开 **cmd** 或 **PowerShell**，进入项目根目录（示例 `E:\paddleOcr`）：

```bat
cd /d E:\paddleOcr
python -m venv .venv
call .venv\Scripts\activate.bat
```

锁定清单 `offline_bundle\wheels\requirements-lock.txt` 已包含 **`paddlepaddle` 及其 CPU 运行所需传递依赖**（如 `protobuf`、`opt-einsum`、`networkx`、`safetensors` 等）。内网安装时只要 `offline_bundle` 中的 `.whl` 齐全，**无需访问外网**。

**推荐（一条命令，双目录查找）**：

```bat
pip install --no-index --find-links offline_bundle\paddle --find-links offline_bundle\wheels -r offline_bundle\wheels\requirements-lock.txt
```

**等价写法（先装 Paddle 栈，再装锁定清单；与旧文档一致）**：

```bat
pip install --no-index --find-links offline_bundle\paddle paddlepaddle==3.3.1
pip install --no-index --find-links offline_bundle\paddle --find-links offline_bundle\wheels -r offline_bundle\wheels\requirements-lock.txt
```

第二步务必同时加上 `--find-links offline_bundle\paddle`，以便解析仅在 `paddle` 目录中的传递依赖 wheel。

若 `requirements-lock.txt` 中版本与你的离线 wheel 不一致，以**你们外网执行 `scripts\prepare_offline_assets.ps1` 打包时锁定的版本**为准，或在外网更新锁定清单后重新 `pip download` 再拷贝。

验证：

```bat
python -c "import fastapi, paddle; print('ok')"
```

## 5. 启动微服务

根目录提供 **三种** 启动脚本，均调用 **uvicorn 多进程**，并默认 `PADDLE_PDX_DISABLE_MODEL_SOURCE_CHECK`、`FLAGS_use_mkldnn`；**默认 `OCR_WORKERS=5`**。请按算力与精度需求选择其一：

| 脚本 | 检测 / 识别模型 | 适用场景 |
|------|-----------------|----------|
| `startupV4m.bat` | `PP-OCRv4_mobile_det` + `PP-OCRv4_mobile_rec` | 资源紧、追求速度；精度低于 v5 |
| `startupv5m.bat` | `PP-OCRv5_mobile_det` + `PP-OCRv5_mobile_rec` | v5 mobile，速度与精度折中 |
| `startupv5s.bat` | `PP-OCRv5_server_det` + `PP-OCRv5_server_rec` | 默认高精度；**内存占用最大**，多 worker 时更易 OOM |

下文以 **`startupv5s.bat`** 为例；换用其它脚本时只需替换文件名。

### 5.1 手动启动（测试）

在项目根目录双击或在命令行执行：

```bat
cd /d E:\paddleOcr
startupv5s.bat
```

上述方式为**前台**启动：当前窗口会输出日志，按 **Ctrl+C** 可停止服务。

### 5.2 后台启动（可选）

若希望**不占用当前命令行窗口**（服务在**新开的最小化窗口**中运行，本窗口执行完即返回），可用以下任一方式：

```bat
cd /d E:\paddleOcr
startupv5s.bat bg
```

或（与 `bg` 等价）：

```bat
set OCR_BACKGROUND=1
startupv5s.bat
```

- 不同脚本对应不同窗口标题（如 `PaddleOCR-v5s`）；停止时关闭该窗口，或在任务管理器中结束对应 **Python** 进程。
- 远程桌面 / 服务器上临时测试时，用后台方式可避免误关当前会话窗口导致误以为服务已停。

可选环境变量（在**同一**命令行窗口中先 `set` 再运行启动脚本）：

| 变量 | 含义 | 示例 |
|------|------|------|
| `OCR_WORKERS` | uvicorn 工作进程数 | 脚本默认 **5**；`startupv5s.bat` 若出现 `Child process died` 可降为 **1～2** 或换 mobile 脚本 |
| `OCR_HOST` | 监听地址 | 默认 `0.0.0.0` |
| `OCR_PORT` | 监听端口 | 默认 `8000` |
| `OCR_BACKGROUND` | 设为 `1` 时与 `xxx.bat bg` 相同，后台（最小化窗口）启动 | `set OCR_BACKGROUND=1` |
| `PADDLE_PDX_DISABLE_MODEL_SOURCE_CHECK` | 设为 `1` 可避免内网下联网检查卡住（各启动脚本默认已设） | 一般无需改 |

健康检查：

- 浏览器或 `curl` 访问：`http://127.0.0.1:8000/health`  
- 期望返回：`{"success":true,"status":"ok"}`

业务接口示例：

- `POST http://服务器IP:8000/v1/ocr/document/idcard`（JSON：`imageBase64` 等，见 `README.md` / C# 示例）

### 5.3 模型路径

默认使用项目内相对路径（无需配置）：

- `offline_bundle\models\PP-OCRv5_server_det_infer`
- `offline_bundle\models\PP-OCRv5_server_rec_infer`

（由 `scripts/download_models.py` 下载的 PaddleX 推理包，内含 `inference.yml` 等。）若模型搬到其他盘符，可设置环境变量 `OCR_DET_MODEL_DIR` / `OCR_REC_MODEL_DIR` 指向**绝对路径**后再运行所选启动脚本。

## 6. 配置开机自动启动（任务计划程序）

推荐用 Windows **任务计划程序** 在系统启动后自动运行所选启动脚本（例如 `startupv5s.bat`）。

### 6.1 基本步骤

1. 打开 **任务计划程序** → **创建任务**（不要用“创建基本任务”，以便勾选最高权限）。
2. **常规**：
   - 名称：`OCRMicroservice`（可自定义）
   - 勾选 **不管用户是否登录都要运行**（视安全策略二选一）
   - 勾选 **使用最高权限运行**
3. **触发器** → 新建 → **启动时**。
4. **操作** → 新建：
   - **程序或脚本**：`cmd.exe`
   - **添加参数**：`/c "E:\paddleOcr\startupv5s.bat"`（若希望计划任务触发后以**最小化窗口**后台跑服务，可改为 `/c "E:\paddleOcr\startupv5s.bat bg"`；按需将 `startupv5s.bat` 换成 `startupv5m.bat` 或 `startupV4m.bat`）
   - **起始于**：`E:\paddleOcr`（与项目根目录一致）
5. **条件**：按需取消“仅交流电源”（笔记本场景）。
6. **设置**：可勾选 **如果任务失败，重新启动**（间隔如 1 分钟，尝试 3 次）。

### 6.2 验证

重启服务器 → 等待数十秒 → 访问 `http://127.0.0.1:8000/health`。

---

## 7. 与 IIS 的关系

本微服务是**独立 HTTP 进程**（默认 `8000` 端口）。IIS 上的业务站点可通过 **反向代理**（ARR + URL Rewrite）将某路径转发到 `http://127.0.0.1:8000`，或由 C# 后端**直接 HTTP 调用**本机 `127.0.0.1:8000`（见 `CSharp-IIS-调用示例.md`）。

## 8. 常见问题

1. **启动很慢或卡住**  
   - 确认已设置 `PADDLE_PDX_DISABLE_MODEL_SOURCE_CHECK=1`（各启动脚本已默认设置）。

2. **`pip install` 报找不到包 / 解析失败**  
   - 确认 `--find-links` 同时包含 `offline_bundle\wheels` 与 `offline_bundle\paddle`（见第 4 节）；Python 主版本与 wheel 文件名中的 **`cp3xx`** 一致；`requirements-lock.txt` 为 **UTF-8**。  
   - 若清单中新增依赖（如 `paddlepaddle` 相关传递包）但未拷贝对应 `.whl`，须在外网补全后重新拷贝（见第 3 节）。

3. **拷贝到内网前如何自检离线包是否完整**  
   - 详见 **第 3.3 节**。简要：在外网执行 `python scripts\verify_offline_bundle.py`，或运行 `scripts\prepare_offline_assets.ps1`（收尾自动校验）。失败则重新执行准备脚本，补全 `offline_bundle\wheels`、`offline_bundle\paddle` 与模型目录后再拷贝。

4. **端口被占用**  
   - `set OCR_PORT=8080` 后重新执行所选启动脚本，并同步修改 IIS / C# 中的调用地址。

5. **任务计划运行失败**  
   - 检查“起始于”是否为项目根目录；运行账户是否有权读取 `offline_bundle`；`.venv` 是否已在该机器上创建并完成 `pip install`。

6. **uvicorn 日志反复出现 `Child process died`、或测试站报 `ReadError`**  
   - 多为 **多 worker 各加载一套 OCR 模型导致内存不足**，子进程被系统结束。请 **`set OCR_WORKERS=1`**（或 **2**）后重启，或改用 **`startupv5m.bat` / `startupV4m.bat`**；内存非常充裕时再酌情调高 worker 数。
